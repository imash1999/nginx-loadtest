Nginx Load Balancing — нагрузочное тестирование и мониторинг

Проект демонстрирует горизонтальное масштабирование веб-приложения с помощью Nginx,
мониторинг через Prometheus и Grafana, оркестрацию через Airflow и нагрузочное
тестирование через Locust. Всё запускается в Docker.


Стек

- Nginx — балансировщик нагрузки, распределяет запросы между воркерами (least_conn)
- FastAPI — три инстанса приложения на портах 8001, 8002, 8003
- PostgreSQL — основная база данных
- Locust — генератор нагрузки, симулирует сотни пользователей
- Prometheus — сбор метрик с каждого воркера
- Grafana — визуализация метрик в реальном времени
- Airflow — оркестрация, запускает тесты по расписанию
- Docker Compose — весь стек в одной команде


Как запустить

Требования: Docker Desktop, 6 GB RAM.

    git clone https://github.com/imash1999/nginx-loadtest
    cd nginx-loadtest
    cp .env.example .env
    docker compose up --build -d

Сервисы после запуска:

    http://localhost       — приложение через Nginx
    http://localhost/stats — распределение запросов по воркерам
    http://localhost:3001  — Grafana (admin / admin)
    http://localhost:9090  — Prometheus
    http://localhost:8080  — Airflow (admin / admin)


Как провести нагрузочный тест

Запустить тест с 200 пользователями на 60 секунд:

    docker compose --profile loadtest run --rm locust locust \
      --host=http://nginx --headless \
      -u 200 -r 50 --run-time 60s \
      -f locustfile.py

Посмотреть распределение запросов между воркерами:

    curl http://localhost/stats


Что показывают графики

В Grafana настроены две панели.

Первая — время ответа 95й перцентиль по каждому воркеру. Во время нагрузочного
теста видны пики, после которых система стабилизируется — это результат балансировки.

Вторая — количество запросов в секунду по воркерам. Три линии идут параллельно
и на одном уровне — Nginx равномерно распределяет нагрузку между app1, app2, app3.

Когда один воркер отключается, его нагрузка автоматически переходит на оставшиеся.
Когда воркер возвращается — нагрузка снова выравнивается.


Airflow DAG

DAG с именем nginx_loadtest_pipeline запускается каждый час и выполняет три шага:
проверяет здоровье всех воркеров, запускает нагрузочный тест, проверяет равномерность
распределения запросов между воркерами и выводит отчёт.

Интерфейс Airflow: http://localhost:8080


Результаты тестирования

Один воркер под нагрузкой 300 пользователей:
- Среднее время ответа: 1013 ms
- Максимум: 1399 ms
- req/s: ~300

Три воркера (Nginx балансировка), та же нагрузка:
- Среднее время ответа: 140 ms
- Максимум: 350 ms
- req/s: ~600

Балансировка снижает время ответа в 7 раз при той же нагрузке.
